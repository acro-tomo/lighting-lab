# photometric-sim 開発記録

各ステップの完了時に「何を実装し、何で検証したか」と設計判断（採用案／却下案）を記録する。

## 前提の設計判断（実装開始時）

- **配置**: 既存アプリ（雰囲気比較シミュレーター）とはプロダクト要件が相反する（既存は「実照度lux・IESを保証しない」が不変条件）ため、リポジトリ内の独立サブプロジェクト `photometric/` として構築。既存 `src/` は無変更。／却下案: 既存アプリの改修 — 不変条件と直接衝突するため。
- **間取り入力形式**: 多角形頂点列[m]＋天井高＋吹抜けポリゴン（部分天井高）の TypeScript/JSON 構造。Phase 1 はコード内サンプルLDKを既定とする。／却下案: 画像取込・作図UI — Phase 1 スコープ外。
- **プリセット初期データ**: スキーマは実在品番前提（品番・メーカー・配光の出所を保持）。初期データは `dataSource: 'representative'`（代表値サンプル）としてUIで明示し、実在カタログ値の確認が取れたものだけ `'catalog'` に昇格させる。架空の値を実在品番として提示しない。
- **対応ブラウザ下限**: WebGL2 必須。WebGPU は利用可能なら自動採用（three.js WebGPURenderer のフォールバック機構を利用）。
- **UIフレームワーク**: React 不使用、vanilla TS + 軽量DOM。仕様がフレームワークとして Three.js のみを指定しており、UI規模も小さいため。
- **描画/測光の関数共有の解釈**: IESサンプリングは「candela格子データ＋補間規則」を単一モジュールに置き、CPU測光はその関数を直接呼ぶ。GPU描画は同一の Float32Array を R32Float テクスチャとして参照し同じ双線形補間でサンプルする（シェーダとCPUで同一関数オブジェクトの共有は物理的に不可能なため、「単一のデータソース＋同一の補間規則」で二重化を防ぐ）。遮蔽は「同一の遮蔽ジオメトリ集合」を測光レイキャストと描画シャドウの両方が使う。

## Step 1–2: データモデル＋測光計算コア（描画なし）

**実装**
- `src/core/types.ts` — lm/cd/lx を型で分離した光源・部屋・家具・マテリアルモデル。発光面寸法（`apertureDiameter`）は IES に含まれないため独立フィールド。
- `src/core/room.ts` — 多角形幾何（面積・内包判定・壁セグメント・吹抜け天井高）。
- `src/core/color.ts` — CCT→xy（Kang 2002 Planck軌跡近似）→XYZ→linear sRGB、輝度=1 正規化（lm維持）。sRGB→linear 反射率変換。
- `src/photometry/distribution.ts` — 配光インターフェース。ビーム角近似 I0 = lm/(2π(1−cos(半角)))、エッジは半角中心±15%のsmoothstep（光束ほぼ保存）。等方配光。数値積分ユーティリティ。
- `src/photometry/illuminance.ts` — E = Σ I(θ,φ)·d·V·cosθ/r²。遮蔽は `OcclusionTester` インターフェース注入。間接光は `IndirectIlluminanceProvider` の差し込み口を確保（Phase 2互換、現状は常に0）。

**検証**: `npm test` 24件パス / `npm run typecheck` クリーン
- 1000cd 直下2mで250lx、逆二乗、余弦則、調光線形、遮蔽0/0.5、裏面0
- lm→cd の半角/全角取り違えガード、ビーム配光の光束保存（数値積分 <2%）
- CCT変換の輝度正規化（=1）、6500K≒D65、B/R単調性、sRGB→linear既知値

**設計判断**
- ビーム配光のエッジ: 採用 = 半角を中心とした対称smoothstep（ピーク値は公式どおり、光束もほぼ保存）／却下 = ハードカットオフ（描画時のジャギー）・cosineべき乗（ピークとlmの関係が公式から乖離）。
- 測光コアは three.js 非依存の純TS。遮蔽レイキャストはアダプタとして描画側モジュールが実装し注入する。
