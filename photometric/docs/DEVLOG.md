# photometric-sim 開発記録

各ステップの完了時に「何を実装し、何で検証したか」と設計判断（採用案／却下案）を記録する。

## 前提の設計判断（実装開始時）

- **配置**: 既存アプリ（雰囲気比較シミュレーター）とはプロダクト要件が相反する（既存は「実照度lux・IESを保証しない」が不変条件）ため、リポジトリ内の独立サブプロジェクト `photometric/` として構築。既存 `src/` は無変更。／却下案: 既存アプリの改修 — 不変条件と直接衝突するため。
- **間取り入力形式**: 多角形頂点列[m]＋天井高＋吹抜けポリゴン（部分天井高）の TypeScript/JSON 構造。Phase 1 はコード内サンプルLDKを既定とする。／却下案: 画像取込・作図UI — Phase 1 スコープ外。
- **プリセット初期データ**: スキーマは実在品番前提（品番・メーカー・配光の出所を保持）。初期データは `dataSource: 'representative'`（代表値サンプル）としてUIで明示し、実在カタログ値の確認が取れたものだけ `'catalog'` に昇格させる。架空の値を実在品番として提示しない。
- **対応ブラウザ下限**: WebGL2 必須。WebGPU は利用可能なら自動採用（three.js WebGPURenderer のフォールバック機構を利用）。
- **UIフレームワーク**: React 不使用、vanilla TS + 軽量DOM。仕様がフレームワークとして Three.js のみを指定しており、UI規模も小さいため。
- **描画/測光の関数共有の解釈**: IESサンプリングは「candela格子データ＋補間規則」を単一モジュールに置き、CPU測光はその関数を直接呼ぶ。GPU描画は同一の Float32Array を R32Float テクスチャとして参照し同じ双線形補間でサンプルする（シェーダとCPUで同一関数オブジェクトの共有は物理的に不可能なため、「単一のデータソース＋同一の補間規則」で二重化を防ぐ）。遮蔽は「同一の遮蔽ジオメトリ集合」を測光レイキャストと描画シャドウの両方が使う。

## Step 1–2: データモデル＋測光計算コア（描画なし）

**実装**
- `src/core/types.ts` — lm/cd/lx を型で分離した光源・部屋・家具・マテリアルモデル。発光面寸法（`apertureDiameter`）は IES に含まれないため独立フィールド。
- `src/core/room.ts` — 多角形幾何（面積・内包判定・壁セグメント・吹抜け天井高）。
- `src/core/color.ts` — CCT→xy（Kang 2002 Planck軌跡近似）→XYZ→linear sRGB、輝度=1 正規化（lm維持）。sRGB→linear 反射率変換。
- `src/photometry/distribution.ts` — 配光インターフェース。ビーム角近似 I0 = lm/(2π(1−cos(半角)))、エッジは半角中心±15%のsmoothstep（光束ほぼ保存）。等方配光。数値積分ユーティリティ。
- `src/photometry/illuminance.ts` — E = Σ I(θ,φ)·d·V·cosθ/r²。遮蔽は `OcclusionTester` インターフェース注入。間接光は `IndirectIlluminanceProvider` の差し込み口を確保（Phase 2互換、現状は常に0）。

**検証**: `npm test` 24件パス / `npm run typecheck` クリーン
- 1000cd 直下2mで250lx、逆二乗、余弦則、調光線形、遮蔽0/0.5、裏面0
- lm→cd の半角/全角取り違えガード、ビーム配光の光束保存（数値積分 <2%）
- CCT変換の輝度正規化（=1）、6500K≒D65、B/R単調性、sRGB→linear既知値

**設計判断**
- ビーム配光のエッジ: 採用 = 半角を中心とした対称smoothstep（ピーク値は公式どおり、光束もほぼ保存）／却下 = ハードカットオフ（描画時のジャギー）・cosineべき乗（ピークとlmの関係が公式から乖離）。
- 測光コアは three.js 非依存の純TS。遮蔽レイキャストはアダプタとして描画側モジュールが実装し注入する。

## Step 3: 直接光レンダリング＋プリセットJSON

**実装**
- `render/renderer.ts` — WebGPURenderer（WebGPU非対応環境はWebGL2へ自動フォールバック、`?backend=webgl2` で強制可）。PBR Neutral、固定露出（EVオフセットのみ）、sRGB出力。レンダラーは `RendererHandle` 抽象の背後（将来差し替え可能）。
- `render/lights.ts` — 全光源（ビーム近似含む）を「CPU測光と同一の LightDistribution からサンプル生成した減衰プロファイルテクスチャ × ピークcd」の IESSpotLight として描画。描画/測光の配光二重化を排除。
- `render/sceneBuilder.ts` — 多角形→床/壁/天井生成（吹抜け: 天井穴＋上部天井＋立ち上がり壁）。家具は表示メッシュと遮蔽メッシュを分離（遮蔽メッシュがシャドウと測光レイキャストを担い、colorWrite=false で描画に不寄与）。
- `render/occlusion.ts` — シャドウキャスターと同一ジオメトリ集合へのレイキャストによる OcclusionTester 実装。
- `app/presets.ts` — プリセットJSONの境界検証。`public/presets.json` は代表値サンプル6種（dataSource=representative、UIで明示）。
- `src/main.ts` — vanilla TS UI。器具リスト/プリセット切替/調光/位置/チルト・パン編集、ドラッグ移動、露出スライダ、補助環境光トグル（1灯のみ・照度計算対象外と明示）、CRI免責表示。

**検証**: typecheck / build パス。headless Chromium（SwiftShader, WebGL2）で描画確認 — 室内に暖色光だまり・遮蔽・吹抜けボリュームを目視確認。コンテナ環境ではWebGPUアダプタ取得後にデバイスロストする（実GPUなし）ため、WebGPU経路の実機確認は今後ユーザー環境で行う。

**設計判断**
- three r184 の IESSpotLightNode は θのみの1Dプロファイル（φ非対応）。GPU側配光はφ平均の軸対称近似とし、測光側は完全な θ/φ を評価する（住宅用DL/スポットはほぼ軸対称。制限としてここに記録）。
- スポットのGPU減衰はコーン角に依存しない iesMap 経路に一本化（angle はシャドウカメラ用のみ）。

## Step 4: 照度ヒートマップ＋実数lxプローブ

**実装**
- `photometry/grid.ts` — グリッド照度計算（0.15m間隔、部屋外・家具内部マスク、回転家具対応）、固定カラースケール（自動正規化なし）。純TSでテスト可能。
- `app/heatmap.ts` — グリッド→CanvasTexture→水平面メッシュ、凡例。スケール 0〜300 / 0〜500 lx 切替。
- `render/overlayMaterial.ts` — **重要な発見**: three のWebGPU系レンダラーはフレーム全体をHDRターゲットに描き出力パスで一括トーンマッピングするため `material.toneMapped=false` が存在しない（ビルドに参照ゼロ）。対策として、オーバーレイ色に 1/toneMappingExposure を乗じるTSLノードを使用（出力パスの×露出で相殺、PBR Neutral は0.8未満で恒等写像なので色が保存される）。毎フレーム露出と同期するため EV を変えてもヒートマップ色は不変＝「露出やトーンマッピングを変えても lx 表示は一切変化しない」を保証。
- クリックで任意点の実数lx表示（グリッド補間でなく測光コアの直接評価）＋マーカー表示。計算面高さ・スケールはUIで変更可。

**検証**: `npm test` 28件パス（グリッドの手計算250lx一致・家具マスク・回転内包・固定スケールのクランプ）。headless確認: 平均82lx/最大1063lx、吹抜け下プローブ92lx（手計算 820lm・60°・4.45m直下×2灯 ≈ 49×2 と整合）。ヒートマップの上下反転（CanvasTexture flipY）を発見し修正、キッチン集光DLの高照度リング・ダイニング約150-200lx・スポットの壁際ビームが位置・強度とも妥当なことを目視確認。

**設計判断**
- ヒートマップ計算はメインスレッド同期＋120msデバウンス（本シーン規模で〜90ms）。将来グリッド密度/部屋規模が増えたらWorker化。
- プローブはグリッド値の参照でなく測光コアの直接評価（「実数lx値」の要件に対応）。
